# Article Structure Examples

Step 4 写作时的骨架参考。**只提供每节的目的、开头句、收尾句和转场**，正文要现场基于真实新闻撰写，不要套模板生搬硬套。

**重要约束**：本文件只抽象热门科技公众号常见的内容结构和阅读节奏，绝不复刻任何具体公众号的标题句式、固定口头禅、栏目包装或表达风格。写作时要像一个有判断力的科技作者，而不是模仿某个账号。

## 通用开篇钩子公式

不要用「近日」「大家好」「值得关注的是」开头。三选一：

### 钩子 A：实战对比（首选）

```
先讲一个让圈内人最近反复转发的对比实验。

{某团队 / 某研究者} 在 {具体场景} 下，把 {A 方案} 和 {B 方案} 并排跑了一次，
唯一变量是 {核心差异}。

{A 的表现，突出失败或挣扎，具体到时间或次数}。

{B 的表现，突出一次就过或效率领先}。

这不是孤例。{引用 benchmark 或第三方数据}。
```

### 钩子 B：反直觉数据

```
{具体数字}。这是 {主角} 在 {基准} 上比 {参照物} 高出的幅度。

如果你熟悉 {基准} 的历史曲线，你会知道每爬 1 个点意味着什么：
{解释该基准的困难度}。而 {主角} 一次吃掉了 {N} 个百分点。

这篇文章要回答三个问题：{问题 1}、{问题 2}、{问题 3}。
```

### 钩子 C：名场面切入

```
{具体场景，一两句话把人带入}。

{角色的一句引述 / 一个动作 / 一个决定}。

这个瞬间背后，是 {新闻核心事件} 正在发生。
```

## 爆款易读通用结构（默认 readability_mode=viral_readable）

无论选哪个 `content_profile`，默认先套这层阅读节奏，再填入具体内容画像。

```
[先说结论] — 80-150 字
  - 3-5 条短判断：这是什么、为什么重要、谁受影响、该不该行动
  - 每条只讲一个判断，不展开论证

[故事钩子 / 场景钩子] — 150-250 字
  - 一个真实使用场景、商业冲突、反直觉数据或人物动作
  - 不用“近日”“随着 AI 发展”

[这和你有什么关系] — 150-250 字
  - 至少点名 2 类读者：开发者 / 产品经理 / 投资人 / 普通用户 / 创业者
  - 明确受益、受冲击或需要改变判断的地方

[主体三段式]
  1. 发生了什么：事实、时间线、产品/交易/论文基本面
  2. 为什么重要：技术、商业、生态、用户行为里的核心变量
  3. 该怎么判断：对不同读者的决策建议

[行动建议 / 后续观察]
  - 谁应该马上试
  - 谁可以观望
  - 接下来盯哪 2-3 个信号
```

每 600-800 字插入一个节奏点：小标题、短表、信息卡、反问、金句或“读者决策提示”。不要连续 6 段都只是平铺解释。

## 内容画像骨架

### `tech_deep_dive`：技术深度型

适用：模型、论文、benchmark、架构升级、推理/多模态/coding 能力突破。

```
先说结论：3-5 条技术判断

一、先把结论翻译成人话：这次到底变强在哪
  - 用一张小表列能力变化
  - 每个术语第一次出现必须解释

二、关键技术拆解：不要只背参数
  - 机制 / 架构 / 训练或推理方式
  - 连续两段硬核解释后接白话类比

三、Benchmark 怎么看：哪些数据能信，哪些只是热闹
  - 正文只放 3-5 行核心表
  - 大表、公式、复现细节进“延伸阅读”

四、落到真实场景：开发者/产品/企业会怎么用
  - 2-4 个具体任务场景
  - 每个场景给“适合 / 不适合”判断

五、结论：这不是分数，而是能力边界变化
```

### `business_narrative`：商业叙事型

适用：并购、战略投资、创始人动态、大厂合作、组织调整、财报里的 AI 变量。

```
先说结论：3-5 条商业判断

一、一个动作背后的信号：从人物/交易/资本动作切入
  - 谁做了什么
  - 为什么是现在

二、真正的矛盾：不是钱，而是控制权/入口/算力/生态
  - 把事件放进行业背景
  - 至少写清 2 方利益

三、谁受益，谁被挤压
  - 表格：参与方 / 得到什么 / 失去什么
  - 不做空泛“具有重要意义”

四、接下来会发生什么
  - 短期 1-3 个月
  - 中期 6-12 个月
  - 最大不确定性

五、给读者的判断框架
```

### `product_experience`：产品体验型

适用：OpenAI Image 2、Cursor vs Claude、AI app、开发者工具、MCP servers、agent skills、GitHub 热门工具、消费级 AI 功能、产品对比。

```
先说结论：谁该用、谁不用急、最大变化是什么

一、用一个真实场景开头
  - 同一个任务，A 产品怎么做，B 产品怎么做
  - 不堆功能名，先讲体验差异

二、核心对比：别问谁更强，先问你要完成什么任务
  - 表格：场景 / 更适合谁 / 原因 / 注意事项
  - 只保留读者会拿来决策的维度

三、三个最容易误解的点
  - 纠正常见认知偏差
  - 每点配一个具体例子

四、上手建议：从哪个场景开始试
  - 个人用户
  - 开发者
  - 团队/企业

五、决策矩阵：现在用、等一等、不要用
```

### `developer_ecosystem`：开发者生态 / 社区趋势型

适用：热门 MCP / MCP servers、Cursor skills、Claude skills、GitHub trending 工具、agent workflow、新工程术语（如 harness engineering / context engineering）。

```
先说结论：这是真趋势、工具热度，还是旧概念换名

一、它为什么突然被讨论
  - GitHub / HN / 官方 repo / 作者博客 / 媒体报道里分别出现了什么信号
  - 明确信号强度：多源趋势 / 社区热议 / 单点冒泡

二、先把概念讲人话
  - MCP / skills / harness engineering 解决的具体问题
  - 用一个开发者日常场景解释：接工具、跑测试、评估 agent、复用工作流

三、对工具链的影响
  - 对个人开发者
  - 对团队工程流程
  - 对平台和插件生态

四、现在该不该跟
  - 表格：适合尝试 / 可以观望 / 暂时别碰
  - 写清风险：标准未定、生态碎片、维护成本、概念炒作

五、后续观察信号
  - repo star / fork / release 频率
  - 主流工具是否集成
  - 是否出现稳定案例和反面复盘
```

### `fast_take`：热点速评型

适用：政策、突发安全、争议传闻、小道融资、人事变动、信息还不完整但热度很高的事件。

```
先说结论：3 点判断 + 1 个不确定性

一、事实边界：已经确认了什么
  - 只列有来源支持的事实
  - 单源消息明确标注

二、为什么市场会关注
  - 这件事触动了哪个旧矛盾
  - 为什么不是普通小新闻

三、现在能下的判断
  - 判断 1：短期影响
  - 判断 2：谁最敏感
  - 判断 3：后续验证信号

四、风险提示：哪些还不能写死
```

### `personal_practice`：第一人称实战复盘型（仅技术层面）

适用：第一人称"我"的技术 / 工程 / 产品方法论复盘，例如「我用 Cursor 半年的实战心得」「我们怎么搭了一套 Harness Engineering 体系」「我做 AI Agent 工程化踩过的 7 个坑」。

**前置约束**：
- 必须经过 Step 3.1 二选一确认（用户选了"是"才能用第一人称"我"）
- 全文 ≥5 处具体到文件名 / 函数名 / 命令参数 / 数字的真实细节
- ≥3 个独立块呈现的可复制资产（prompt / 命令 / 配置 / checklist / 目录树 / 决策矩阵 / 取舍表）
- 必出失败迭代模块

```
[自我介绍 + 转折点] — 约 200 字
  - 我是谁（行业 / 角色 / 年限）
  - 我以前做不了什么、现在能做什么（建立信任和共鸣）
  - 这段经历让我对哪个问题有了切身体会，也是写本文的出发点

[先说结论] — 约 150 字
  - 3-5 条短判断：核心方法论是什么、谁该跟、谁不必跟

一、{方法论 / 工具 / 范式} 的三次跃迁（或其他演化框架） — 约 600 字
  - 时代 / 阶段对比表（4-7 行 × 4-6 列）
  - 我自己在每个阶段的真实状态
  - 用一段话讲清楚"现在到了哪一步"

二、一个真实案例：{主线项目名} 的进化之路 — 约 800 字
  2.1 项目背景（200 字）：项目是什么、为什么做、用什么写的
  2.2 阶段 A：粗暴起步 → 痛点显而易见
  2.3 阶段 B：第一次升级 → 立竿见影但新问题
  2.4 阶段 C：现在的解法 → 关键转变是什么
  - 每个阶段必须写"痛点 → 解 → 新痛点"
  - 必须有具体到文件名 / 命令 / 数字的细节

三、搭建 {方法论或体系}：具体做了什么 — 约 1000 字
  3.1 子模块 1：为什么不能这样做 / 现在怎么做
    - 表格：旧方案 vs 新方案
    - 真实小故事：那次它发现 / 抓到 / 帮我避开了什么
  3.2 子模块 2：迭代过程
    - 第一版的致命问题
    - 迭代一 / 二 / 三：每次改了什么、为什么改
  3.3 子模块 3：实战怎么启动
    - 三步指南
    - 可复制的命令 / prompt / 目录结构

四、事后验证 / 安全网 — 约 400 字
  - 为什么这一步如此重要
  - 验证清单 / 检查项表格
  - 一份真实的报告样例

五、轻重取舍：什么任务用什么强度 — 约 250 字
  - 表格：任务类型 → 建议流程 → 说明
  - 关键原则：流程重量与任务重量匹配

六、给同行的清单 — 约 250 字
  - 立刻能抄的 5 件事
  - 可以观望的 3 件事
  - 别一上来就做的 2 件事

[结语] — 约 100 字
  - 一句话总结：如果只能记一句话

参考资料 — 3-5 条链接（含原始论文 / 官方文章 / 工具仓库 / 自己仓库的公开部分）
```

### `retrospective`：项目 / 产品迭代复盘型（仅技术层面）

适用：技术产品 / 工程项目 / 团队工作流从 V1 → Vn 的迭代复盘，例如「我们的 Agent 体系从 V1 到 V12 的进化史」「这套 verify_all.ps1 是怎么从 1010 行砍到 275 行的」。

**前置约束**：
- 必须经过 Step 3.1 二选一确认
- 必出版本时间线表
- 必出"V1 致命问题 + 迭代一/二/三"模块

```
[项目背景] — 约 200 字
  - 项目是什么、解决什么问题
  - 团队规模、技术栈
  - 为什么值得复盘（受众有多大、踩坑代价有多重）

[先说结论] — 约 150 字
  - 3-5 条复盘判断：哪些做对了 / 哪些走了弯路 / 哪些经验可以直接抄

一、版本时间线一张图 — 约 250 字
  - 表格：版本号 / 上线日期 / 核心改动 / 当时的痛点
  - 一段话定基调："这个项目从第一行代码起就是 X，到现在经历了 Y 个大版本，是 Z 的试验田"

二、V1 的致命问题：什么被低估了 — 约 400 字
  - 当初的设计假设
  - 哪个假设崩了
  - 崩的代价（数字 / 时间 / 返工量）

三、迭代一：定义 {核心机制} — 约 400 字
  - 解决方案
  - 关键决策的取舍
  - 留下的隐患

四、迭代二：把每个 {子模块} 的操作流程拆细 — 约 400 字
  - 第一版的粗放：只规定做 X，没说 X 包括哪些动作
  - 现在的细化：审查维度 / 操作流程 / 验收标准
  - 表格：旧版有 vs 新版有

五、迭代三：{隐性问题或跨任务记忆} — 约 400 字
  - 这是个"不起眼但实际影响很大"的改进
  - 解决了什么持续性问题
  - 给跨会话 / 跨团队 / 跨周期带来了什么

六、收获与教训 — 约 350 字
  - 5 件做对了的事
  - 3 件早该做但拖了的事
  - 1 件如果重来不会再做的事

七、给后来者的避坑清单 — 约 250 字
  - 表格：场景 → 推荐做法 → 反例
  - 是否值得复制：哪些团队 / 哪些项目适合直接照搬，哪些需要改造

参考资料
```

## 新模型发布文模板（3000-3800 字）

参考 `C:\Users\Administrator\claude-opus-4-7-article.md`（Claude Opus 4.7 那篇）。完整章节骨架：

```
[开篇钩子 A] — 约 250 字

一、{主角} 发布速览：先把基本面交代清楚 — 约 300 字
  - 发布时间 / 可用渠道 / 定价 / 上下文 / 视觉 / 定位（列表形式）
  - 两个值得圈出来的点（价格策略 / 能力定位）
  收尾句："这里有两个值得圈出来的点。"

二、Benchmark 全景对比：为什么这次是质变 — 约 600 字
  - 数据表格（4-7 行 × 5 列）
  - 关键 benchmark 的单独解读（3 段）
  - 对照竞品的横向位置
  收尾句："这也解释了为什么 {主角} 把核心叙事押在 {X} 上。"

三、四大能力跃迁：把 benchmark 数字翻译成开发者语言 — 约 900 字
  3.1 {能力 1}：{一句话概括效果}
  3.2 {能力 2}：{一句话概括效果}
  3.3 {能力 3}：{一句话概括效果}
  3.4 {能力 4}：{一句话概括效果}
  每个能力 200-250 字，结构固定：现象描述 → 数据或案例 → 工程意义

四、行为变化与迁移指南：把 4.X 的 prompt 直接搬过来会踩坑 — 约 450 字
  - 警示色块包装的破坏性变化
  - 3 条具体的迁移建议（编号列表）
  收尾句："简言之，{主角} 把 {X} 从 A 推向了 B。"

五、安全与合规升级 — 约 300 字（若有）
  - 官方自动拦截机制
  - 合规通道（认证项目）
  - 对应用层的工程意义

六、决策矩阵：何时升级，何时留在旧版 — 约 350 字
  - 通过色块：强烈建议升级的场景（5 条）
  - 警示色块：可以暂缓的场景（4 条）
  - 一句话过渡策略

七、结语：在 frontier 竞争中的位置 — 约 200 字
  - 纯推理 / 编码 / agentic 三条赛道的位置
  - 竞品差异化
  - 居中金句收尾

参考资料 — 5-7 条链接
免责声明 — 数据时间戳提醒
```

## 5 个备选标题的通用公式

每篇文章最后都要出 5 个备选标题，**必须覆盖 5 种类型**：

| 类型 | 公式 | 示例 |
| --- | --- | --- |
| 吸睛实证 | `{主角}实测：{场景 A}，{数据 B}，{数据 C}` | "Claude Opus 4.7 实测：同一个 Bug，4.6 卡了 45 分钟，4.7 用了 11 分钟" |
| 数据冲击 | `{主角}发布：{关键数字}，{戏剧化结论}` | "Opus 4.7 发布：SWE-bench Pro 暴涨 11 个点，Anthropic 把编码王座拿回来了" |
| 专业判断 | `从{A}到{B}：{主角}这次不是{小事}，而是{大事}` | "从 4.6 到 4.7：Claude 这次不是小版本，而是 Agentic 编码的代际跃迁" |

## 视觉风格与文章类型匹配表（Step 4.5 参考）

不同类型的文章在章节节奏、表格占比、强调块频次上有不同偏好。Step 4.5 选 `component_variant` 时把这张表和 [style-template.md](style-template.md) Section D 对照着看。

| `news_type` | 章节数 | 表格数 | 强调块数 | 章节标题风格偏好 | 引导块风格偏好 |
| --- | --- | --- | --- | --- | --- |
| `flagship_release`（旗舰发布） | 5-7 | 1-2（必须有 benchmark 表） | 2-3（迁移指南 + 决策矩阵） | `giant-numeral` / `left-bar`（信息量大，需要清晰编号） | `giant-quote`（开篇钩子有张力） |
| `paper_breakthrough`（论文突破） | 4-6 | 1（实验数据表） | 0-1 | `symmetric-dash` / `small-caps`（学术克制） | `meta-line` / `center-caption`（安静进入） |
| `industry_event`（行业大事件） | 4-5 | 0-1（时间线可用表） | 1-2（多方观点对照） | `divider-double` / `bracket-prefix`（新闻质感） | `bordered-card`（头版感） |
| `developer_ecosystem`（开发者生态） | 4-5 | 1（场景/决策表） | 2（概念解释 + 风险边界） | `left-bar` / `giant-numeral`（工具链路径清晰） | `bordered-card` / `gray-bar`（结论前置） |
| `rumor_funding`（小道消息） | 3 | 0 | 0-1 | `left-bar` / `dotted-prefix`（轻盈） | `center-caption`（一句话带入） |
| `practice_methodology`（个人实战复盘） | 5-8 | 2-4（时代对比表 + 取舍表 + 检查项 + 决策矩阵） | 3-5（失败迭代块 + 资产卡 + 关键洞察块） | `giant-numeral` / `bracket-prefix`（自传质感 + 编号清晰） | `giant-quote`（自我介绍开场张力） |
| `product_evolution`（项目 / 产品迭代复盘） | 4-6 | 1-3（版本时间线表 + 取舍表 + 避坑清单） | 2-3（V1 致命问题块 + 迭代节点块） | `divider-double`（新闻档案感）+ 时间或版本标签（"Prompt 时代 V3.0~V3.3"） | `bordered-card` / `meta-line`（结论前置 + 元信息感） |

**匹配逻辑**：文章结构密度 × 视觉元素密度 应当同向。旗舰发布字数多、章节多、数据多，视觉上就允许用 `giant-numeral` / `card-rows` 这种高密度组件；小道消息字数少，视觉也要轻，用 `left-bar` + `minimal-bottom` 表格即可。

- [ ] 没有"小编"/"笔者"/"咱们"
