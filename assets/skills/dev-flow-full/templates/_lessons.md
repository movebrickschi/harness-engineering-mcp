## 沉淀分类说明

每次复盘按以下分类追加：

- **PM 沟通踩坑**：哪些问题应该早问？哪些理解差异花了大代价？
- **B 项目隐性规范**：未文档化但团队默认遵守的约定
- **A 与 B 的差异模式**：A/B 间反复出现的差异类型
- **架构决策**：选 X 不选 Y 的权衡过程
- **复用机会**：可被未来需求复用的能力
- **踩过的坑**：技术坑 + 解决方案
- **耗时经验**：估算偏差及原因

## 示例条目（首次使用可删除）

## [2026-04-21] supplier-batch-approve

### PM 沟通踩坑
- "批量审核失败要不要全部回滚"这个问题没早问，开发完才发现 PM 想要"部分成功"，返工 2 小时
- 教训：批量操作类需求，启动前必须确认"全有或全无"还是"尽力而为"

### B 项目隐性规范
- 所有 admin 接口都要在 audit_log 表留记录（团队默认，无文档）
- DTO 文件统一放在 `dto/` 子目录，不要平铺
- 测试文件命名 `*.spec.ts` 而非 `*.test.ts`

### A 与 B 的差异
- A 用 Express 中间件做权限，B 用 NestJS Guard 装饰器
- A 用 sequelize，B 用 prisma → 关联查询写法差异大

### 架构决策
- 审核日志选择独立 ApprovalLog 表而非在 Supplier 加 history JSON 字段
  - 理由：未来要支持按操作者/时间范围查询，独立表索引更方便

### 复用机会
- 新建的 BatchActionMixin 可复用到其他批量操作场景（订单批量发货等）
- audit log 装饰器化后续可以独立 PR 推广

### 踩过的坑
- Prisma 的 batch update 不返回更新数量 → 用 updateMany + count 替代
- 前端 useBatchSelect 在表格刷新后会丢失选中状态 → 加 key 持久化

### 耗时经验
- 估算 12.5h，实际 18h
- 超时主要原因：批量操作的边界场景没在方案阶段考虑充分（占 4h）
