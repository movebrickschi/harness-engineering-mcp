## ✨ 新增功能

### 🗂️ `.harness/` 统一目录布局

旧版本里散落在 4 个位置的 harness 产物现在全部收拢：

```
project-root/
├── .harness/                              ← 统一目录（新）
│   ├── config.json                        # 原 harness.config.json
│   ├── baseline.json                      # 原 verification_baseline.json
│   ├── engineering-harness.md             # 原 docs/engineering-harness.md
│   ├── scripts/engineering-check.{ps1,sh} # 原 项目根
│   ├── adr/0001-...md                     # 原 docs/adr/
│   ├── features/INDEX.md + _template/     # 原 docs/features/
│   └── oncall.md | SLO.md | DORA.md | ... # mid-team/org 升档产物
├── CHANGELOG.md                           ← 保留在根（npm 约定）
└── .github/CODEOWNERS · pull_request_template.md  ← 保留（GitHub 约定）
```

**好处**：
- 一眼能看出"这是个用 harness 的项目"
- 不污染 docs/（团队自己的文档不会和 harness 产物混在一起）
- 卸载时一个目录搞定（见 `harness_uninstall`）

### 🗑️ `harness_uninstall` — 第 7 个 MCP 工具

完整卸载 harness 一行命令：

```bash
harness uninstall --dry-run    # 先预览要删什么
harness uninstall              # 交互确认后递归删除 .harness/
harness uninstall -y           # 跳过确认（脚本场景）
harness uninstall --keep-root-dir -y   # 清空内部但保留 .harness/ 占位
```

**`CHANGELOG.md` 和 `.github/*` 自动保留**（因 npm / GitHub 等外部工具约定），列在返回值 `kept[]` 提醒用户手工处理。

支持 MCP 工具调用：

```jsonc
harness_uninstall {
  "cwd": "/path/to/project",
  "dry_run": true,
  "keep_root_dir": false
}
```

### 🛡️ `harness init --force` — 安全默认 + 显式覆盖

**0.1.x 行为（危险）**：同 `harness init` 跑第二遍，已存在但内容不同的文件**默默被模板覆盖**——你写在 `INDEX.md` 里的任务条目 / 手写 ADR / 调过的覆盖率阈值，**全没**。

**0.2.0 默认行为（安全）**：

| 文件状态 | v0.1.x | **v0.2.0 默认** |
|---|---|---|
| 不存在 | created | created |
| 存在且内容 = 模板 | skipped(unchanged) | skipped(unchanged) |
| 存在且内容 ≠ 模板 | **updated（覆盖）⚠️** | **skipped(kept_existing)** ✅ |

**显式覆盖**用 `--force` 开关：

```bash
harness init --force --mode=solo --stack=node-typescript --type=library --name=demo
```

> ⚠️ `--force` 会覆盖 `.harness/features/INDEX.md` 等可能含真实任务记录的文件。**重新生成前务必 git commit 一次**。

## ⚠️ Breaking Changes

### 1. 目录布局硬切换（**没有自动 fallback**）

0.1.x 的老项目升级到 0.2.0 后，`harness check` 会因找不到 `.harness/config.json` 报 FAIL。**必须迁移**。

### 2. `harness init` 默认行为

如果你的 CI / 脚本里有"重跑 init 把所有文件刷新一遍"的逻辑，0.2.0 默认**不再覆盖**，会被报为 `skipped(kept_existing)`。如需保持老行为，加 `--force`。

## 🔄 迁移指南

### 方案 A（推荐 · 保留 customizations）

```powershell
# 1. 备份
git add -A
git commit -m "backup before .harness/ migration"

# 2. 用 git mv 搬迁（保留 git 历史）
git mv harness.config.json     .harness/config.json
git mv verification_baseline.json .harness/baseline.json
git mv docs/engineering-harness.md .harness/engineering-harness.md
git mv docs/adr                .harness/adr
git mv docs/features           .harness/features
mkdir .harness/scripts -ErrorAction SilentlyContinue
git mv engineering-check.ps1   .harness/scripts/
git mv engineering-check.sh    .harness/scripts/

# 如果用过 mid-team / org 升档：
git mv docs/oncall.md .harness/oncall.md
git mv docs/SLO.md    .harness/SLO.md
git mv docs/DORA.md   .harness/DORA.md
git mv docs/SBOM.md   .harness/SBOM.md
git mv docs/rfc       .harness/rfc
git mv compliance     .harness/compliance

# 3. 验证
harness check
git commit -m "migrate to .harness/ layout (v0.2.0)"
```

### 方案 B（重置 · 丢 customizations）

适合**没修改过任何 harness 产物**的项目：

```powershell
# 删老文件
Remove-Item harness.config.json, verification_baseline.json, engineering-check.* -Force
Remove-Item -Recurse docs/adr, docs/features
# 重 init
harness init -C .
```
